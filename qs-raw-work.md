# Findings

# Startale Stablecoin Vaults Security Audit

## Info
* Auditor: Andrei Stefan
* Commit hash: 4eb3a96
* Repository: -- 
* Documentation quality: Medium

**Disclaimer**
This audit covers the EarnVault system including RewardRedistributor, EarnVaultUpgradeable, EarnVaultStorageBase, BoostRewardsLib, and EarnVault contracts. The audit focuses on yield distribution mechanisms, index-based accounting, access control, upgrade safety, and boost reward distribution. The audit does not cover the underlying M0 USDSC token implementation or external oracle/price feed systems. Critical issues related to carry accounting, blacklist mechanics, and array growth have been identified and should be addressed before production deployment.

## Key Actors and their Capabilities

1. **Users (Depositors)**: Can deposit USDSC tokens into EarnVault to earn yield. Users' yield accrues via an index-based accounting system with RAY (1e27) precision. Users can withdraw their principal at any time along with automatically claimed yield. Users can explicitly claim accrued USDSC yield and boost rewards (ASTR, DOT, etc.) without withdrawing principal. Users can utilize gasless deposits via `depositWithPermit()` if USDSC supports EIP-2612 permits. Users are subject to blacklist restrictions - blacklisted users cannot deposit, withdraw, or claim.

2. **Yield Redistributor (Contract)**: The RewardRedistributor contract is the authorized caller of `onYield()` on EarnVault. It pulls freshly-minted USDSC yield from M0 extension contract via `claimYield()`. It applies configurable fee (0-20%) to gross yield, sending fee to treasury. It distributes net yield proportionally to two cohorts: EarnVault users (checkbox OFF) and sUSDSC vault holders (checkbox ON). Distribution is based on each cohort's share of total USDSC supply before the mint. Uses per-cohort integer carry accumulators to minimize long-run rounding bias. Any remainder yield (from wallets/LP/rounding) goes to Startale treasury.

3. **Admin Roles (Owner)**: Controls via Ownable2Step pattern in both upgradeable and non-upgradeable versions. Can set critical addresses: yieldRedistributor, boostRewardKeeper, treasury, and pauser. Can modify blacklist status of any address, blocking their access to all vault functions. Can trigger emergency recovery via `recoverERC20()` when contract is paused. Can sweep surplus USDSC to treasury above minimum required reserves. Can sweep accidentally sent native ETH via `sweepNative()`. For upgradeable version, controls proxy upgrades with upgrade authority.

4. **Operator Role (RewardRedistributor)**: Granted `OPERATOR_ROLE` in RewardRedistributor contract. Can call `distribute()` to pull and redistribute yield from M0 extension. Must call this function regularly to distribute accrued yield to vault users. Calling frequency impacts user yield realization but not total yield allocation.

5. **Boost Reward Keeper**: Separate privileged address authorized to call `onBoostReward()`. Distributes external reward tokens (ASTR, DOT, etc.) to vault users. Must transfer tokens to vault before calling `onBoostReward()`. Rewards distributed proportionally based on user principals, similar to USDSC yield. Uses same RAY precision index accounting as USDSC yield.

6. **Pauser**: Dedicated address authorized to pause/unpause the vault contract. Pause blocks all user operations: deposit, withdraw, claim, and boost reward distribution. Pause does NOT block owner administrative functions like recovery and surplus sweeping. Pause does NOT block yield distribution from redistributor. Separate role allows emergency response without full owner privileges.

7. **Treasury**: Receives protocol fees from yield distribution in RewardRedistributor. Receives all yield when vault has zero deposits (totalPrincipal == 0). Receives boost rewards when vault has zero deposits. Receives swept surplus USDSC when vault balance exceeds required reserves. Does not require special permissions - is simply a recipient address.

## Operational Considerations

1. **Index-Based Yield Accounting with RAY Precision**: The vault uses a global index scaled to RAY (1e27) for high-precision yield distribution. When yield arrives via `onYield()`, the index increases by `(amount * RAY) / totalPrincipal`. Each user has a personal index `userIndex` tracking when they last settled. User's pending yield is `principal * (globalIndex - userIndex) / RAY`. This eliminates per-user storage updates on yield distribution but requires settlement before balance-changing operations. The system uses a `_carryRay` accumulator for remainder tracking: `_carryRay = (amount*RAY + _carryRay) % totalPrincipal` to ensure exact yield distribution over time with no permanent rounding bias.

2. **Two-Step Yield Distribution Flow**: Yield distribution requires exact sequencing: (1) External caller (RewardRedistributor) transfers USDSC to vault, (2) Caller invokes `onYield(amount)`, (3) Vault verifies balance >= claimReserve + amount, (4) Vault updates globalIndex and claimReserve. If step 1 is skipped or amount is incorrect, the funding check fails and reverts. This prevents index inflation without actual backing funds. The same pattern applies to boost rewards via `onBoostReward()`.

3. **Claim Reserve Funding Invariant**: The vault maintains `claimReserve` tracking the total amount users can claim: principals + accrued yield. On deposit, claimReserve increases by deposit amount (1:1 principal reservation). On yield distribution, claimReserve increases by yield amount. On withdrawal/claim, claimReserve decreases by paid amount. The contract enforces `USDSC.balanceOf(vault) >= claimReserve` at all times. Any violation causes `InsufficientFunding` revert. Operator must ensure yield is actually transferred before calling `onYield()`, or the invariant check will fail.

4. **Blacklist Interaction with Existing Positions**: Users can deposit while not blacklisted, then get blacklisted afterward. Blacklisted users cannot call deposit, withdraw, claim, or interact with the vault in any way. Their principal and accrued yield remain locked in the contract with no user-side recovery mechanism. Owner can use `recoverERC20()` when paused to recover tokens, but this requires manual intervention. View functions like `getClaimableBoostReward()` explicitly check blacklist status and revert, preventing even read operations.

5. **Boost Rewards Distribution Architecture**: Boost rewards (external tokens like ASTR, DOT) use the same index-based accounting as USDSC yield. Each token has separate `boostGlobalIndex[token]` and `boostClaimReserve[token]`. Users have `userBoostIndex[user][token]` and `userBoostAccrued[user][token]` mappings. The `activeBoostTokens` array tracks all tokens ever distributed for enumeration. Unlike USDSC yield, boost rewards do NOT use carry accounting - remainder is lost. Boost rewards are automatically claimed during withdrawal but must be explicitly claimed otherwise.

6. **Upgradeable Storage Pattern (ERC7201)**: EarnVaultUpgradeable uses ERC7201 namespaced storage to prevent collisions. All state variables stored in single `EarnVaultStorage` struct at deterministic slot `keccak256(abi.encode(uint256(keccak256("startale.storage.EarnVault")) - 1)) & ~bytes32(uint256(0xff))`. This ensures upgrade-safe storage layout. The constructor calls `_disableInitializers()` to prevent implementation contract initialization. Initialization happens via `initialize()` on proxy deployment. The storage struct includes fields for both USDSC yield (principal, globalIndex, accrued) and boost rewards (boostGlobalIndex, userBoostAccrued, activeBoostTokens).

7. **RewardRedistributor Allocation Logic**: RewardRedistributor calculates yield split based on total USDSC supply BEFORE the mint (`S_base = totalSupply - minted`). EarnVault TVL is `T_earn = earnVault.totalPrincipal()`. sUSDSC vault TVL is `T_yield = susdscVault.totalAssets()`. After fee deduction: `toEarn = (net * T_earn + carryEarn) / S_base`, `toOn = (net * T_yield + carryOn) / S_base`. Remainder `net - (toEarn + toOn)` goes to treasury. This ensures users only receive yield proportional to their share of circulating supply, excluding tokens not in either vault (wallets, LP, etc.).

8. **First Depositor Considerations**: When totalPrincipal is 0 and first user deposits, their userIndex is set to current globalIndex (which starts at RAY). If significant yield is distributed immediately after with tiny totalPrincipal, the index can increase dramatically: `delta = (largeYield * RAY) / tinyPrincipal`. This creates large index values but doesn't break accounting. However, subsequent precision in small yield distributions could be affected. No explicit first depositor protection or minimum deposit requirement exists.

9. **Emergency Recovery Mechanisms**: Owner can call `recoverERC20()` to rescue tokens, but only when paused for USDSC. For USDSC, only surplus above `claimReserve` can be recovered. For other tokens (including boost tokens), only amounts above `boostClaimReserve[token]` can be recovered. Native ETH can be swept anytime via `sweepNative()` since vault should never hold ETH. Surplus USDSC can be swept to treasury via `sweepSurplusToTreasury()` even when not paused, but this is limited to amount above claimReserve.

10. **Pausability Scope and Limitations**: Pause blocks user functions: deposit, depositWithPermit, withdraw, claim, and boost reward distribution. Pause does NOT block administrative functions: setters, recovery, surplus sweeping. Pause does NOT block `onYield()` - the redistributor can still push yield even when users cannot interact. This allows yield accumulation during pause without disrupting external distribution mechanisms. Users can claim accumulated yield once unpaused.

11. **Boost Token Array Growth**: The `activeBoostTokens` array grows every time a new token is distributed via `onBoostReward()`. Once added, tokens are never removed from the array even if their balance reaches zero. Every withdraw() and claim() operation loops through all active boost tokens to settle and claim rewards. As the array grows, gas costs for these operations increase linearly. No maximum array size or pruning mechanism exists.

12. **Carry Accounting Split**: In RewardRedistributor, the `distribute()` function calculates yield split via `_calculateSplit()` which is a view function returning what the carry values should be. Then it manually updates the carry storage variables. The calculation and state update are separated. If `_calculateSplit()` logic differs from manual carry update logic (lines 167-171), carry values could drift. Currently they use identical logic but the separation creates maintenance risk.

## Issues

### [High] Leveraged TVL Manipulation via Sandwich Attack on distribute()

**Description**
The RewardRedistributor's `distribute()` function reads the sUSDSC vault's `totalAssets()` at execution time to calculate yield splits. An attacker can build a leveraged position in the sUSDSC vault using a lending pool (e.g., Aave-style), sandwich the `distribute()` transaction, and capture disproportionate yield while stealing from EarnVault users. The attack uses borrowed USDSC as collateral in recursive loops to artificially inflate sUSDSC's TVL, manipulating the yield distribution formula.

```solidity
// RewardRedistributor.sol:324 - Uses spot TVL (manipulable)
T_yield = susdscVault.totalAssets(); // ← Read at transaction execution time

// RewardRedistributor.sol:335-342 - Yield split calculation
if (useCarries) {
    uint256 numEarn = net * T_earn + _carryEarn;
    toEarn = numEarn / S_base;  // EarnVault allocation
    
    uint256 numOn = net * T_yield + _carryOn;
    toOn = numOn / S_base;  // sUSDSC allocation (inflated if T_yield manipulated)
}
```

**Exploit Scenario:**
Assuming sUSDSC vault has 10M USDSC, EarnVault has 5M principal, and distribute() is called every 3-4 hours with ~1M USDSC yield. Attacker uses 1M USDSC capital with access to a lending pool that accepts sUSDSC collateral at 90% LTV.

1. Attacker detects operator's `distribute()` transaction in mempool
2. Attacker frontruns with leverage loop (5 iterations): Deposit 1M USDSC → get 1M sUSDSC → borrow 900k USDSC. Deposit 900k → get 900k sUSDSC → borrow 810k. Continue until 4.095M total deposited using 3.685M borrowed + 1M original capital. This inflates sUSDSC vault from 10M to 14.095M USDSC.
3. Operator's `distribute()` executes with manipulated TVL:
   - Normal split would be: toEarn = 316,667 USDSC, toOn = 633,333 USDSC
   - Manipulated split: toEarn = 248,752 USDSC (lost 67,915), toOn = 701,248 USDSC (gained 67,915)
   - Attacker's 4.095M shares (29% of vault) capture 203,693 USDSC of the inflated yield
4. Attacker backruns: Redeem 4.095M shares for 4,298,693 USDSC, repay 3.685M loan (interest ~1.4 USDSC for 2 minutes), net profit 203,191 USDSC per attack
5. Attack repeatable 6 times daily = 1.2M USDSC profit/day = 445M USDSC profit/year (44,493% APY)

The vulnerability exists because ERC4626's share accounting is fair - attacker gets correct shares at deposit and correct assets at redemption. The exploit is in `distribute()` using spot TVL instead of time-delayed snapshot, allowing same-block manipulation.


**Files(s)**
- src/distributor/RewardRedistributor.sol 
- src/vaults/4626/SUSDSCVaultUpgradable.sol 

**Recommendation**
Implement snapshot-based TVL mechanism to prevent same-block manipulation:

```solidity
// In RewardRedistributor.sol
uint256 public lastSUSDSCTVL;
uint256 public lastSnapshotBlock;

function snapshotTVL() external onlyRole(OPERATOR_ROLE) {
    lastSUSDSCTVL = susdscVault.totalAssets();
    lastSnapshotBlock = block.number;
}

function distribute() external whenNotPaused onlyRole(OPERATOR_ROLE) nonReentrant {
    require(lastSnapshotBlock == block.number - 1, "Must snapshot in previous block");
    // Use lastSUSDSCTVL instead of susdscVault.totalAssets() in calculations
    // ... rest of function
}
```

Operator workflow: Call `snapshotTVL()` in block N, then call `distribute()` in block N+1. This forces attacker to hold leveraged position for minimum 1 full block (~12 seconds), making attack 100x more expensive.

For defense-in-depth, add 1-hour withdrawal cooldown to sUSDSC vault:

```solidity
// In SUSDSCVaultUpgradable.sol
uint256 public constant WITHDRAWAL_COOLDOWN = 1 hours;
mapping(address => uint256) public lastDepositTime;

function deposit(uint256 assets, address receiver) public override returns (uint256) {
    uint256 shares = super.deposit(assets, receiver);
    lastDepositTime[receiver] = block.timestamp;
    return shares;
}

function withdraw(uint256 assets, address receiver, address owner) public override returns (uint256) {
    require(block.timestamp >= lastDepositTime[owner] + WITHDRAWAL_COOLDOWN, "Cooldown active");
    return super.withdraw(assets, receiver, owner);
}
```

### [High] Boost Rewards Not Settled During Deposit Operations

**Description**
The `deposit()` and `depositWithPermit()` functions settle USDSC yield via `_settle()` but do NOT settle boost rewards via `_settleBoost()`. This creates an accounting vulnerability where users can increase their principal without updating their boost reward indices, causing them to earn boost rewards on their new principal retroactively from their old index position. This allows users to steal boost rewards from other depositors.

```solidity
// EarnVaultUpgradeable.sol:194-207 - deposit() function
function deposit(uint256 amount) external virtual whenNotPaused nonReentrant {
    EarnVaultStorage storage $ = _getStorage();
    _checkNotBlacklisted(msg.sender);
    if (amount == 0) revert IEarnVaultEventsAndErrors.ZeroAmount();

    _settle(msg.sender);  // ← Only settles USDSC yield, NOT boost rewards!

    $.USDSC.safeTransferFrom(msg.sender, address(this), amount);
    $.principal[msg.sender] += amount;  // ← Principal increased without settling boost
    $.totalPrincipal += amount;
    $.claimReserve += amount;

    emit Deposit(msg.sender, amount);
}
```

Compare with `withdraw()` which DOES settle boost rewards:
```solidity
// EarnVaultUpgradeable.sol:249-272 - withdraw() function
function withdraw(uint256 amount) external virtual whenNotPaused nonReentrant {
    EarnVaultStorage storage $ = _getStorage();
    _checkNotBlacklisted(msg.sender);
    if (amount == 0) revert IEarnVaultEventsAndErrors.ZeroAmount();

    _settle(msg.sender);  // Settles USDSC yield

    uint256 p = $.principal[msg.sender];
    if (amount > p) revert IEarnVaultEventsAndErrors.InsufficientPrincipal();

    // Settle and claim ALL boost rewards in single loop
    for (uint256 i = 0; i < $.activeBoostTokens.length; i++) {
        address token = $.activeBoostTokens[i];
        _settleBoost(msg.sender, token);  // ← Settles boost rewards properly!
        BoostRewardsLib.claimBoostReward(/*...*/);
    }
    // ... rest of function
}
```

Boost reward indices track "yield per unit of principal" using the formula `owed = principal * (boostGlobalIndex - userBoostIndex) / RAY`. When a user deposits without settling boost rewards, their `userBoostIndex` remains at the old value while their `principal` increases, causing the calculation to apply the index delta to ALL their principal (old + new), not just the old principal that was actually deposited during that period.

**Exploit Scenario**
Assuming boost rewards are distributed regularly:

1. **Time T0 - Initial State:**
   - Alice deposits 1000 USDSC (first deposit)
   - `principal[Alice] = 1000`
   - `userBoostIndex[Alice][TokenA] = 0` (default for new user)
   - `boostGlobalIndex[TokenA] = 0`

2. **Time T1 - First Boost Distribution:**
   - Keeper distributes 1000 TokenA boost rewards to vault
   - `boostGlobalIndex[TokenA]` increases: `0 + (1000 * 1e27) / 1000 = 1e27`
   - Alice's pending rewards: `1000 * (1e27 - 0) / 1e27 = 1000 TokenA` ✅

3. **Time T2 - Alice Deposits More WITHOUT Settlement:**
   - Alice calls `deposit(1000)` (doubles her position)
   - `_settle()` is called → USDSC yield settled ✅
   - **`_settleBoost()` is NOT called** → Boost indices NOT updated ❌
   - Alice's state:
     - `principal[Alice] = 2000` (doubled!)
     - `userBoostIndex[Alice][TokenA] = 0` (STILL 0! Not updated to 1e27)
   - Alice's 1000 TokenA is still pending in `userBoostAccrued`

4. **Time T3 - Second Boost Distribution:**
   - Keeper distributes another 1000 TokenA
   - Total distributed: 2000 TokenA in the system
   - `boostGlobalIndex[TokenA]` increases: `1e27 + (1000 * 1e27) / 2000 = 1.5e27`

5. **Time T4 - Alice Tries to Claim:**
   - Alice calls `claim()` or `withdraw()`
   - `_settleBoost()` is called
   - **Incorrect calculation:**
     - `owed = 2000 * (1.5e27 - 0) / 1e27 = 3000 TokenA`
   - **Alice tries to claim 3000 TokenA when only 2000 exist!**
   - Transaction REVERTS with `InsufficientBoostClaimReserve()`

6. **Correct Calculation Should Be:**
   - Old principal (1000): earned T0→T1: `1000 * 1e27 / 1e27 = 1000 TokenA`
   - Old principal (1000): earned T1→T3: `1000 * 0.5e27 / 1e27 = 500 TokenA`
   - New principal (1000): earned T2→T3: `1000 * 0.5e27 / 1e27 = 500 TokenA`
   - **Total correct: 2000 TokenA**

**Recommendation**
Add boost reward settlement to both deposit functions:

```solidity
function deposit(uint256 amount) external virtual whenNotPaused nonReentrant {
    EarnVaultStorage storage $ = _getStorage();
    _checkNotBlacklisted(msg.sender);
    if (amount == 0) revert IEarnVaultEventsAndErrors.ZeroAmount();

    _settle(msg.sender);
    
    // ADD: Settle all boost rewards before changing principal
    for (uint256 i = 0; i < $.activeBoostTokens.length; i++) {
        _settleBoost(msg.sender, $.activeBoostTokens[i]);
    }

    $.USDSC.safeTransferFrom(msg.sender, address(this), amount);
    $.principal[msg.sender] += amount;

##### 2 of 8: Boost Rewards Not Settled During Deposit Operations

**Severity:** High

**Description:**
````

#### Summary

**Operational Considerations:**
````
1. **Index-Based Yield Accounting with RAY Precision**: The vault uses a global index scaled to RAY (1e27) for high-precision yield distribution. When yield arrives via `onYield()`, the index increases by `(amount * RAY) / totalPrincipal`. Each user has a personal index `userIndex` tracking when they last settled. User's pending yield is `principal * (globalIndex - userIndex) / RAY`. This eliminates per-user storage updates on yield distribution but requires settlement before balance-changing operations. The system uses a `_carryRay` accumulator for remainder tracking: `_carryRay = (amount*RAY + _carryRay) % totalPrincipal` to ensure exact yield distribution over time with no permanent rounding bias.

2. **Two-Step Yield Distribution Flow**: Yield distribution requires exact sequencing: (1) External caller (RewardRedistributor) transfers USDSC to vault, (2) Caller invokes `onYield(amount)`, (3) Vault verifies balance >= claimReserve + amount, (4) Vault updates globalIndex and claimReserve. If step 1 is skipped or amount is incorrect, the funding check fails and reverts. This prevents index inflation without actual backing funds. The same pattern applies to boost rewards via `onBoostReward()`.

3. **Claim Reserve Funding Invariant**: The vault maintains `claimReserve` tracking the total amount users can claim: principals + accrued yield. On deposit, claimReserve increases by deposit amount (1:1 principal reservation). On yield distribution, claimReserve increases by yield amount. On withdrawal/claim, claimReserve decreases by paid amount. The contract enforces `USDSC.balanceOf(vault) >= claimReserve` at all times. Any violation causes `InsufficientFunding` revert. Operator must ensure yield is actually transferred before calling `onYield()`, or the invariant check will fail.

4. **Blacklist Interaction with Existing Positions**: Users can deposit while not blacklisted, then get blacklisted afterward. Blacklisted users cannot call deposit, withdraw, claim, or interact with the vault in any way. Their principal and accrued yield remain locked in the contract with no user-side recovery mechanism. Owner can use `recoverERC20()` when paused to recover tokens, but this requires manual intervention. View functions like `getClaimableBoostReward()` explicitly check blacklist status and revert, preventing even read operations.

5. **Boost Rewards Distribution Architecture**: Boost rewards (external tokens like ASTR, DOT) use the same index-based accounting as USDSC yield. Each token has separate `boostGlobalIndex[token]` and `boostClaimReserve[token]`. Users have `userBoostIndex[user][token]` and `userBoostAccrued[user][token]` mappings. The `activeBoostTokens` array tracks all tokens ever distributed for enumeration. Unlike USDSC yield, boost rewards do NOT use carry accounting - remainder is lost. Boost rewards are automatically claimed during withdrawal but must be explicitly claimed otherwise.

6. **Upgradeable Storage Pattern (ERC7201)**: EarnVaultUpgradeable uses ERC7201 namespaced storage to prevent collisions. All state variables stored in single `EarnVaultStorage` struct at deterministic slot `keccak256(abi.encode(uint256(keccak256("startale.storage.EarnVault")) - 1)) & ~bytes32(uint256(0xff))`. This ensures upgrade-safe storage layout. The constructor calls `_disableInitializers()` to prevent implementation contract initialization. Initialization happens via `initialize()` on proxy deployment. The storage struct includes fields for both USDSC yield (principal, globalIndex, accrued) and boost rewards (boostGlobalIndex, userBoostAccrued, activeBoostTokens).

7. **RewardRedistributor Allocation Logic**: RewardRedistributor calculates yield split based on total USDSC supply BEFORE the mint (`S_base = totalSupply - minted`). EarnVault TVL is `T_earn = earnVault.totalPrincipal()`. sUSDSC vault TVL is `T_yield = susdscVault.totalAssets()`. After fee deduction: `toEarn = (net * T_earn + carryEarn) / S_base`, `toOn = (net * T_yield + carryOn) / S_base`. Remainder `net - (toEarn + toOn)` goes to treasury. This ensures users only receive yield proportional to their share of circulating supply, excluding tokens not in either vault (wallets, LP, etc.).

8. **First Depositor Considerations**: When totalPrincipal is 0 and first user deposits, their userIndex is set to current globalIndex (which starts at RAY). If significant yield is distributed immediately after with tiny totalPrincipal, the index can increase dramatically: `delta = (largeYield * RAY) / tinyPrincipal`. This creates large index values but doesn't break accounting. However, subsequent precision in small yield distributions could be affected. No explicit first depositor protection or minimum deposit requirement exists.

9. **Emergency Recovery Mechanisms**: Owner can call `recoverERC20()` to rescue tokens, but only when paused for USDSC. For USDSC, only surplus above `claimReserve` can be recovered. For other tokens (including boost tokens), only amounts above `boostClaimReserve[token]` can be recovered. Native ETH can be swept anytime via `sweepNative()` since vault should never hold ETH. Surplus USDSC can be swept to treasury via `sweepSurplusToTreasury()` even when not paused, but this is limited to amount above claimReserve.

10. **Pausability Scope and Limitations**: Pause blocks user functions: deposit, depositWithPermit, withdraw, claim, and boost reward distribution. Pause does NOT block administrative functions: setters, recovery, surplus sweeping. Pause does NOT block `onYield()` - the redistributor can still push yield even when users cannot interact. This allows yield accumulation during pause without disrupting external distribution mechanisms. Users can claim accumulated yield once unpaused.

11. **Boost Token Array Growth**: The `activeBoostTokens` array grows every time a new token is distributed via `onBoostReward()`. Once added, tokens are never removed from the array even if their balance reaches zero. Every withdraw() and claim() operation loops through all active boost tokens to settle and claim rewards. As the array grows, gas costs for these operations increase linearly. No maximum array size or pruning mechanism exists.

12. **Carry Accounting Split**: In RewardRedistributor, the `distribute()` function calculates yield split via `_calculateSplit()` which is a view function returning what the carry values should be. Then it manually updates the carry storage variables. The calculation and state update are separated. If `_calculateSplit()` logic differs from manual carry update logic (lines 167-171), carry values could drift. Currently they use identical logic but the separation creates maintenance risk.
````

### [High] Price Update Restrictions Enable Stablecoin Depeg Arbitrage

**Description**
The price update mechanism enforces both frequency intervals and deviation boundaries that prevent timely price adjustments during rapid market movements. During stablecoin depegging events, these dual restrictions create extended arbitrage windows where attackers can exploit stale prices.

```solidity
// OracleRegistry.sol:178-181
if (latestPricesUpdatedAt() + paramRegister.getPriceUpdateInterval() > updatedAt)
    revert Errors.PriceUpdateTooFrequent();

// OracleRegistry.sol:171-174
if (_vaultTokenPrice > latestExchangeRate + maxUpperDeviation || 
    _vaultTokenPrice < latestExchangeRate - maxLowerDeviation)
    revert Errors.ExceedMaxDeviation();
```

There are 2 roots for this problem:

1. **Frequency Restriction**: Prevents updates within minimum interval (e.g., 1 hour)
2. **Deviation Boundaries**: Rejects price updates exceeding tolerance (e.g., 10%)

During extreme events, BOTH restrictions block legitimate updates, extending the arbitrage window.

**Exploitation Scenario**

1. USDC market price drops from $1.00 to $0.88 within minutes (12% drop)
2. Protocol's stored USDC price still at $1.00 (last update was 30 minutes ago)
3. Operator attempts emergency update to $0.88:
   - Frequency check: Must wait another 30 minutes (1 hour interval)
   - Even if frequency passed, deviation check: 12% drop exceeds 10% tolerance → reverts
4. Arbitrage window: 30+ minutes (potentially hours if deviation stays high)**
5. Arbitrageurs exploit:
   - Buy USDC on market at $0.88
   - Deposit to protocol at $1.00 oracle price
   - Receive shares worth $1.00 per USDC deposited
   - Request withdrawal in USDT (still at $1.00)
   - Receive USDT worth $1.00
   - Net profit: ~13.6% per transaction
6. Protocol impact: Accepts depegged USDC at inflated prices, drains healthy stablecoin reserves


If deviation boundaries prevent any update, operators must wait for market price to recover closer to stored price OR manually adjust tolerance parameters through governance (multi-day timelock), extending exploitation window indefinitely.

**Files(s)**
- src/oracle/OracleRegistry.sol

**Recommendation**

## Key Components 

### L2UpgradeableERC20 (Implementation Contract):

This is the logic contract for the ERC20 token on Layer 2 (L2). It implements the OptimismMintableERC20 standard & ERC20Upgradable. It contains the core functionality of the token, such as minting, burning, transferring, and querying balances. To be compliant with the OptimismMintableERC20 specifications the burning and minting of tokens needs be performed only by the bridge, in this case the L2UpgradeableERC20 have the variable `BRIDGE` which holds the bridge role, the live contract have been checked to ensure configuration is corect.  


### Transparent Proxy:

This proxy contract holds the actual state of the token contract, including user balances and allowances. It forwards calls  (trough delegatecall ) from users to the implementation contract (L2UpgradeableERC20) while maintaining the state.


### ProxyAdmin Contract:

The ProxyAdmin is a separate contract that has exclusive rights to upgrade the implementation behind the Transparent Proxy. It allows the contract owner to replace the current implementation with a new one (e.g., upgrading the logic for new features or bug fixes) without affecting the state stored in the proxy.


## Key Actors 

### BRIDGE (Mint/Burn Authority): 

The bridge is the only actor that has the authority to mint or burn tokens on the L2. Its role is to manage cross-chain token transfers between Layer 1 and Layer 2.
When tokens are moved from L1 to L2, the bridge will mint the equivalent tokens on L2 for the user.
When tokens are moved from L2 to L1, the bridge will burn the corresponding tokens on L2 before unlocking the tokens on L1.
This ensures that the total supply of tokens across both layers is consistent and reflects the actual balance held by users across the L1 and L2 chains.



### Contract Owner / ProxyAdmin Owner (Upgrade Authority):

The entity (or entities) with control over the ProxyAdmin contract. This actor is responsible for upgrading the logic contract (the L2UpgradeableERC20 contract) by using the ProxyAdmin to point the proxy to a new implementation. This actor can be an individual, a multisig wallet, or a DAO.


## Key Differences between L2UpgradeableERC20 & OptimismMintableERC20

1. L2UpgradeableERC20 does not support the `ILegacyMintableERC20`  interface, compatiblity tess have been runned and there are no security risk or concerns with this difference
1. L2UpgradeableERC20 does not implement the versioning contract `Semver`, compatiblity tess have been runned and there are no security risk or concerns with this difference
1. OptimismMintableERC20 does not implement `permit` function, we could not identiy any security issue or concern with the add of the `permit` functionality, we are especially encouraging and supporting it because it will improve the overall user experience.
